# 题库模拟练习与题库管理设计

## 背景

当前系统已经具备：

- 基于简历的模拟面试主链路；
- 基于 `backend/assets/material/` 的题库/知识库材料目录；
- 基于材料校验、规范化、题库构建、知识库向量化的离线导入脚本；
- 管理端“材料重建”任务页面。

现阶段缺少两类面向业务闭环的能力：

1. 用户无法直接基于现有题库开始“做题式练习”；
2. 管理员无法在网站上直接扩充题库，只能通过仓库内材料文件和重建脚本维护。

本设计新增一个独立于模拟面试的“题库练习”域，并补齐网页端题库管理能力。第一版聚焦“抽题、作答、保存进度、结束练习、扩充题库”，不引入自动评分与总结报告。

## 目标

- 为普通用户提供基于题库直接开始练习的能力；
- 同时支持两类练习模式：
  - 固定题单练习；
  - 可扩展为追问式的练习模式；
- 为管理员提供网页端题库管理能力：
  - Markdown 文件上传；
  - 单题表单录入；
- 所有新增题目必须继续遵循现有材料规范和导入链路；
- 尽量不影响现有模拟面试主链路。

## 非目标

- 第一版不做每题即时点评；
- 第一版不做整场练习总结报告；
- 第一版不做语音输入/语音播报练习；
- 第一版不做自由格式文件上传（如 Excel、CSV、Word）；
- 第一版不做管理员在线编辑已有题目正文。

## 设计决策

### 决策 1：新增独立 `practice` 业务域

题库练习不复用现有 `interview` 会话模型，也不依赖简历、面试阶段或报告流程。

原因：

- 现有 `interview` 域围绕“自我介绍 -> 项目深挖 -> 技术追问 -> 行为追问 -> 结束”的阶段机组织；
- 题库练习需要支持固定题单与可扩展追问两种模式；
- 若直接复用 `interview` 域，会把“简历依赖、阶段依赖、报告依赖”一起带入，后续维护成本高。

因此本次新增独立的 `practice` 会话、题目快照与答案记录表，并提供单独 API、前端状态和页面。

### 决策 2：题库管理继续以材料目录为事实来源

题库管理页无论是“Markdown 上传”还是“单题录入”，都不直接写 `question_bank` 表作为唯一事实源。

统一策略：

1. 先落盘到 `backend/assets/material/` 下对应岗位的题库材料文件；
2. 执行材料校验；
3. 执行规范化；
4. 执行题库构建；
5. 刷新数据库中的 `question_bank`。

原因：

- 与当前仓库规范一致；
- 保证线下维护和线上维护使用同一数据标准；
- 降低线上管理端生成“脏题目结构”的风险；
- 保证题目来源可追溯、材料可审查、脚本可重复执行。

### 决策 3：第一版只做练习过程，不做评分

用户已确认第一版只需要：

- 开始练习；
- 逐题作答；
- 保存过程与进度；
- 查看练习记录；
- 结束练习。

不在本次范围内：

- 自动评分；
- 参考答案打分；
- 薄弱项分析；
- 练习结束总结报告。

## 用户角色

### 普通用户

- 创建练习会话；
- 选择题量、岗位、类别和练习模式；
- 逐题提交文字回答；
- 查看当前进度和已答记录；
- 继续未结束练习；
- 主动结束练习。

### 管理员

- 查看题库列表；
- 按岗位、类别、关键词筛选题目；
- 上传符合规范的 Markdown 题库文件；
- 通过表单录入单题；
- 查看导入/构建异步任务状态与错误信息。

## 用户流程

### 流程 A：固定题单练习

1. 用户进入“题库练习”准备页；
2. 选择岗位、练习模式=`sequence`、题量、类别筛选；
3. 系统从题库中按条件抽取 N 题，创建练习会话；
4. 前端进入练习作答页；
5. 页面一次展示一题，用户输入文字回答后提交；
6. 系统保存本题答案并返回下一题；
7. 所有题目完成后，会话自动变为 `FINISHED`；
8. 用户可查看本次练习记录。

### 流程 B：追问式练习（第一版保留结构）

1. 用户进入“题库练习”准备页；
2. 选择模式=`followup`；
3. 系统先从题库抽首题并创建练习会话；
4. 用户提交文字回答；
5. 后端通过统一的“下一题生成接口”返回下一题；
6. 第一版允许实现为“首题来自题库，后续仍按练习会话统一推进”，但必须保留后续扩展为真正追问逻辑的结构；
7. 结束时仍只保存题目和答案，不生成评分报告。

### 流程 C：管理员上传 Markdown 文件扩充题库

1. 管理员进入题库管理页；
2. 选择岗位并上传 Markdown 文件；
3. 系统校验文件扩展名、内容结构和题目规范；
4. 通过校验后，文件落盘到 `backend/assets/material/` 对应岗位目录；
5. 系统启动异步导入任务；
6. 管理员在页面查看任务状态；
7. 成功后题库列表刷新，失败时展示校验或构建错误。

### 流程 D：管理员单题录入

1. 管理员进入题库管理页；
2. 打开“新增题目”表单；
3. 输入岗位、类别、题干、解析、来源说明等字段；
4. 服务端将表单内容转换为标准 Markdown 结构；
5. 追加写入对应岗位的增量题库文件，或写入新的增量文件；
6. 启动统一导入任务；
7. 成功后题目可在题库列表中查询。

## 领域模型

### 练习会话 `practice_sessions`

建议字段：

- `session_id`
- `user_id`
- `job_role`
- `mode`：`sequence | followup`
- `status`：`ACTIVE | FINISHED`
- `question_count`
- `current_index`
- `category_filters`
- `source_filters`
- `started_at`
- `updated_at`
- `finished_at`

职责：

- 表示一次练习的主状态；
- 保存筛选条件与当前进度；
- 作为题目快照和答案记录的聚合根。

### 会话题目快照 `practice_session_questions`

建议字段：

- `session_question_id`
- `session_id`
- `sequence`
- `question_bank_record_id`
- `job_role`
- `category`
- `title`
- `question`
- `analysis`
- `source_path`
- `snapshot_payload`
- `created_at`

职责：

- 保存本次练习真正抽到的题和顺序；
- 对题目做快照，避免题库变动导致历史练习回放失真；
- 为后续练习记录页提供稳定展示数据。

### 练习答案 `practice_answers`

建议字段：

- `answer_id`
- `session_id`
- `session_question_id`
- `sequence`
- `answer_text`
- `created_at`

职责：

- 保存用户逐题文字回答；
- 支持练习过程回显与结束后回看；
- 为后续若要接入点评/评分预留输入数据。

## 题库类别规范

当前材料中已出现类别文本，如：

- `技术`
- `项目`
- `场景`
- `行为`

但现有规范化产物中的 `category` 大量为空。为支撑题库练习筛选与题库管理，第一版需要显式引入类别标准化。

建议枚举：

- `technical`
- `project`
- `scenario`
- `behavior`

建议映射：

- `技术` -> `technical`
- `项目` -> `project`
- `场景` -> `scenario`
- `行为` -> `behavior`

兼容要求：

- 题库存量数据在规范化时补齐类别映射；
- 管理端表单仅允许选择枚举值，不允许自由输入；
- 页面展示仍使用中文标签。

## API 设计

### 用户练习接口

#### `POST /practice/sessions`

用途：

- 创建练习会话。

请求字段建议：

- `job_role`
- `mode`
- `question_count`
- `category_filters`
- `source_filters`

响应字段建议：

- `session_id`
- `status`
- `mode`
- `current_index`
- `total_questions`
- `current_question`

约束：

- 固定题单练习在创建时一次性抽题并写入快照表；
- 若筛选后题量不足，返回可理解错误。

#### `GET /practice/sessions/{session_id}`

用途：

- 查询当前练习状态、进度和当前题。

响应字段建议：

- `session_id`
- `status`
- `mode`
- `job_role`
- `current_index`
- `total_questions`
- `completed_count`
- `current_question`

#### `POST /practice/sessions/{session_id}/answers`

用途：

- 提交当前题文字答案，并推进到下一题。

请求字段建议：

- `session_question_id`
- `answer_text`

响应字段建议：

- `session_id`
- `status`
- `completed_count`
- `next_question`
- `finished`

约束：

- 仅允许文字回答；
- 会话已结束时禁止继续提交；
- 必须校验提交题目与当前进度一致，避免乱序提交。

#### `POST /practice/sessions/{session_id}/finish`

用途：

- 主动结束会话。

响应字段建议：

- `session_id`
- `status`

#### `GET /practice/sessions/{session_id}/records`

用途：

- 查看本次练习已答记录。

响应字段建议：

- 会话元信息；
- 每题题干；
- 用户答案；
- 题目顺序与完成时间。

### 管理端题库接口

#### `GET /practice/questions`

用途：

- 分页查询题库。

筛选条件建议：

- `job_role`
- `category`
- `keyword`
- `page`
- `page_size`

#### `POST /practice/questions/upload`

用途：

- 上传 Markdown 文件扩充题库。

请求建议：

- `multipart/form-data`
- `job_role`
- `file`

约束：

- 仅允许 `.md`；
- 必须通过结构校验；
- 落盘路径必须限制在材料目录下；
- 写入后触发异步导入任务。

#### `POST /practice/questions`

用途：

- 单题表单录入。

请求字段建议：

- `job_role`
- `category`
- `title`
- `question`
- `analysis`
- `source_note`

约束：

- 服务端负责生成标准 Markdown；
- 严格使用统一模板；
- 写入后走统一导入任务。

#### `GET /practice/questions/import-tasks/{task_id}`

用途：

- 查询上传/录入触发的异步导入任务状态。

## 材料落盘与导入策略

### Markdown 上传

落盘规则：

- Java 题库 -> `backend/assets/material/java/java-interview/`
- Web 题库 -> `backend/assets/material/web/`

文件命名建议：

- 保留原始文件名，但增加冲突保护；
- 若同名文件已存在，生成带时间戳或 UUID 后缀的新文件名。

### 单题录入

建议落盘策略：

- 每次录入写入岗位对应的增量文件；
- 例如：
  - Java：`backend/assets/material/java/java-interview/admin-added-YYYYMMDD.md`
  - Web：`backend/assets/material/web/admin-added-YYYYMMDD.md`

原因：

- 避免直接改写人工维护的原始材料文件；
- 便于追踪哪些题目来自网页管理端；
- 便于后续代码审查和清理。

### 统一导入链路

上传与录入成功后，统一复用现有导入流程：

1. `validate_materials.py`
2. `normalize_materials.py`
3. `build_question_bank.py`

第一版不要求每次新增题目都重建知识库向量索引，除非该题库管理页同时声明会影响知识检索。当前需求只围绕题库练习，因此最小实现应只覆盖题库相关链路。

## 交互要求

### 练习准备页

- 不要求选择简历；
- 默认岗位沿用现有岗位枚举；
- 默认模式可为固定题单练习；
- 若存在未结束练习，优先展示“继续练习”入口。

### 练习作答页

- 一次只展示当前题；
- 只支持文字输入；
- 提交成功后清空输入框并切题；
- 会话结束后跳转至记录页或结束态页面；
- 不展示“实时分”或“报告生成中”等面试专属元素。

### 题库管理页

- 上传前明确展示格式要求；
- 表单录入时类别使用枚举控件；
- 导入任务失败时展示可读错误；
- 成功后自动刷新列表。

## 安全与约束

- 所有 `/practice/sessions*` 接口必须要求普通登录用户；
- 所有 `/practice/questions*` 写接口必须要求管理员；
- 上传文件必须限制扩展名、MIME 类型和落盘目录；
- 禁止路径穿越；
- 用户仅能读取自己的练习会话与记录；
- 管理员录入和上传操作应保留审计日志字段或最小可追踪信息。

## 错误处理

需要覆盖的典型错误：

- 岗位枚举非法；
- 类别枚举非法；
- 筛选后题量不足；
- 会话不存在；
- 无权访问他人会话；
- 会话已结束但继续提交；
- 上传文件不是 Markdown；
- 上传文件结构不符合“第X题 / 题干 / 类别 / 解析”规范；
- 导入任务执行失败；
- 单题录入生成材料文件失败。

## 风险与待确认项

### 风险

- 当前题库规范化产物中类别缺失较多，若不先补齐类别映射，会影响筛选和录入一致性；
- Web 题库当前可能集中在单一文件，单题录入后的增量文件组织方式要保持清晰；
- 追问模式第一版若设计过重，会拖慢交付。

### 待确认项

- 第一版是否在页面上直接暴露“追问模式”，还是先保留后端结构、前端灰度展示；
- 题量上限是否需要限制，例如最大 20 题；
- 单题录入的“来源说明”是否只用于材料注释，还是需要独立入库字段。

## 完成定义（DoD）

- 用户可从网站进入题库练习并完成文字作答；
- 固定题单练习可正常保存进度和记录；
- 追问模式具备可扩展的数据与接口结构；
- 管理员可通过 Markdown 上传和单题录入扩充题库；
- 新增题目仍符合现有材料规范与导入链路；
- 现有模拟面试主流程无回归；
- 接口文档和 Postman 集合同步更新。
