# 开发协作规范

更新时间：2026-06-28

本文用于规范 AI-Interview 小组协作、分支提交、接口联调、测试验证和文档维护。目标是让成员在课程项目周期内减少冲突、保留清晰提交记录，并能在答辩时解释每个人的贡献。

## 1. 协作目标

- 保证功能开发有明确负责人和验收标准。
- 保证接口、页面、数据脚本和文档同步更新。
- 保证每次提交能说明具体改动。
- 保证合并前至少完成基本自测。
- 保证答辩时可以根据提交记录、文档和演示流程说明工作量。

## 2. 推荐角色分工

| 角色 | 主要职责 | 产出 |
| --- | --- | --- |
| 前端负责人 | 页面、路由、状态管理、接口封装 | React 页面、组件、前端测试 |
| 后端负责人 | API、service、repository、鉴权 | FastAPI 路由、业务服务、数据库结构 |
| 数据负责人 | 材料、题库、知识库、向量库 | Markdown 材料、导入脚本、构建报告 |
| 测试负责人 | 测试用例、CI、验收脚本 | pytest、Vitest、Playwright、测试说明 |
| 文档负责人 | 用户手册、部署、答辩材料 | docs 文档、演示脚本、验收清单 |

实际项目中一个人可以承担多个角色，但提交信息应能体现具体职责。

## 3. 分支命名规范

推荐分支格式：

```text
<type>/<scope>-<short-desc>
```

示例：

| 分支 | 说明 |
| --- | --- |
| `feat/interview-schedule` | 新增面试预约 |
| `fix/auth-refresh-loop` | 修复 token 刷新循环 |
| `docs/deployment-runbook` | 补充部署手册 |
| `test/practice-session` | 补充练习流程测试 |
| `chore/update-ci` | 调整 CI 配置 |

如果使用 Codex 辅助开发，可以使用：

```text
codex/<scope>-<short-desc>-<date>
```

## 4. 提交信息规范

推荐使用简洁的 Conventional Commits 风格：

```text
<type>: <summary>
```

常用类型：

| 类型 | 场景 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 缺陷修复 |
| `docs` | 文档修改 |
| `test` | 测试修改 |
| `refactor` | 不改变行为的重构 |
| `chore` | 构建、脚本、依赖等维护 |
| `ci` | CI 配置 |

示例：

```text
feat: add interview schedule calendar export
fix: prevent duplicate report retry requests
docs: expand data and knowledge base guide
test: cover practice session resume flow
```

## 5. 提交粒度

### 5.1 推荐做法

- 一个提交只解决一个清晰问题。
- 文档、代码和测试可以在同一功能提交中一起出现。
- 大功能拆成接口、页面、测试、文档多个提交更容易 review。
- 修复格式化可以单独提交，避免掩盖业务变更。

### 5.2 不推荐做法

- 把多天所有改动压成一个模糊提交。
- 使用 `update`、`final`、`fix bug` 等过于笼统的信息。
- 在功能提交中混入无关重构。
- 提交本地数据库、日志、临时文件或私密配置。

## 6. Pull Request 说明模板

建议每个 PR 写清楚：

```markdown
## 变更内容

- 待填写

## 验证方式

- 待填写

## 影响范围

- 待填写

## 截图或演示

- 待填写

## 风险与后续

- 待填写
```

课程项目即使不强制 PR，也可以把这套结构用于提交说明或答辩记录。

## 7. 前后端接口协作

### 7.1 接口先行

新增业务流程时建议先明确：

- 请求方法。
- URL。
- 请求体。
- 响应体。
- 鉴权要求。
- 错误码。
- 幂等要求。
- 是否异步。

接口确定后同步更新：

- `openapi/openapi.yaml`
- `postman/AI-Interview.postman_collection.json`
- `docs/API接口说明.md`
- 前端 `frontend/src/api/` 封装

### 7.2 联调顺序

1. 后端提供最小可用接口。
2. Postman 或 curl 验证请求。
3. 前端封装 API 方法。
4. 页面接入真实接口。
5. 补充错误态和 loading 态。
6. 写入接口说明和验收用例。

### 7.3 错误处理约定

后端错误应使用统一结构：

```json
{
  "error": {
    "code": "DOMAIN_ERROR_CODE",
    "message": "用户可理解的错误提示"
  }
}
```

前端应调用统一错误解析方法，不在每个页面重复拼接错误文案。

## 8. 前端开发规范

### 8.1 目录约定

| 目录 | 用途 |
| --- | --- |
| `frontend/src/api/` | API client 和业务接口 |
| `frontend/src/pages/` | 页面组件 |
| `frontend/src/components/` | 公共组件 |
| `frontend/src/stores/` | Zustand 状态 |
| `frontend/src/hooks/` | 复用 hook |
| `frontend/src/utils/` | 工具函数 |

### 8.2 页面开发要求

- 页面应处理 loading、empty、error 和 success 状态。
- 写操作按钮应避免重复点击。
- 长耗时任务应展示处理中状态。
- 表单字段需要前端基础校验。
- 鉴权页面不应在未登录状态下请求用户私有数据。

### 8.3 API 调用要求

- 业务页面优先调用 `api/` 中的封装方法。
- 不在页面中散落 baseURL。
- 不手动拼接 token。
- 对写接口保留幂等键能力。
- 401 处理交给统一 client。

### 8.4 测试建议

- 表单校验逻辑写组件或页面测试。
- store 行为写单元测试。
- 关键流程写 Playwright E2E。
- 对工具函数保持轻量、明确的测试。

## 9. 后端开发规范

### 9.1 目录约定

| 目录 | 用途 |
| --- | --- |
| `backend/app/api/` | FastAPI 路由 |
| `backend/app/services/` | 业务服务 |
| `backend/app/repositories/` | 数据访问 |
| `backend/app/models/` | Pydantic 模型 |
| `backend/app/core/` | 配置、安全、错误、日志 |
| `backend/migrations/` | 数据库迁移 |

### 9.2 分层要求

- API 层负责请求解析、鉴权和响应模型。
- service 层负责业务流程。
- repository 层负责 SQLite 访问。
- provider 调用应封装，避免散落在路由中。
- 错误码和异常结构应统一。

### 9.3 数据库修改要求

修改数据库结构时应同步：

- 新增迁移文件。
- 更新 repository 初始化或自修复逻辑。
- 更新数据字典。
- 补充必要测试。
- 考虑旧数据兼容。

### 9.4 异步任务要求

- 长耗时任务不要阻塞主请求。
- 任务状态应可查询。
- 失败时保存错误信息。
- 前端应能重试或展示失败原因。
- 任务执行过程应有日志或 trace ID。

## 10. 数据材料维护规范

### 10.1 材料来源

题库和知识库材料主要位于：

```text
backend/assets/material/
```

运行时构建产物主要位于：

```text
backend/assets/data/
```

### 10.2 修改流程

1. 修改材料 Markdown 或 JSON。
2. 执行材料校验脚本。
3. 执行归一化 dry-run。
4. 构建题库或知识库。
5. 查看 reports 目录中的报告。
6. 更新数据与知识库说明。

### 10.3 材料质量要求

- 题干清楚。
- 解析可独立理解。
- 类别值统一。
- 知识库内容避免重复和过短。
- 编程题样例和判题用例一致。

## 11. 测试协作规范

### 11.1 本地测试

前端常用：

```bash
npm test
npm run lint
npm run test:e2e
```

后端常用：

```bash
pytest
ruff check .
```

具体命令以 `docs/测试与质量说明.md` 和 package 配置为准。

### 11.2 提交前检查

- 是否能启动前端和后端。
- 是否通过相关单元测试。
- 是否检查过核心页面。
- 是否更新接口文档。
- 是否移除调试日志和临时文件。
- 是否避免提交 `.env`、数据库和大体积构建产物。

### 11.3 测试记录

测试负责人可以在 PR 或提交说明中记录：

- 执行命令。
- 测试结果。
- 失败项和原因。
- 未覆盖风险。

不要声称运行了未实际运行的测试。

## 12. 文档维护规范

### 12.1 必须同步更新文档的场景

- 新增页面或用户流程。
- 新增接口或改变请求响应。
- 修改数据表、材料格式或脚本。
- 修改启动方式、端口或环境变量。
- 修改测试命令或 CI 流程。
- 新增管理员操作。

### 12.2 文档位置

| 文档 | 适用场景 |
| --- | --- |
| `docs/README.md` | 文档索引 |
| `docs/API接口说明.md` | 接口变更 |
| `docs/系统架构说明.md` | 架构和模块 |
| `docs/数据与知识库说明.md` | 数据和材料 |
| `docs/部署与运维手册.md` | 启动、部署、环境 |
| `docs/测试与质量说明.md` | 测试和质量 |
| `docs/演示与验收指南.md` | 答辩和验收 |

## 13. 代码 Review 清单

### 13.1 通用项

- 改动是否解决明确问题。
- 是否存在无关修改。
- 是否影响已有功能。
- 是否有错误处理。
- 是否有必要测试。
- 是否更新文档。

### 13.2 前端项

- loading、empty、error 是否完整。
- 表单校验是否清楚。
- token 失效处理是否统一。
- 路由权限是否正确。
- 组件是否存在重复逻辑。

### 13.3 后端项

- 鉴权和用户隔离是否正确。
- 输入校验是否足够。
- SQL 是否参数化。
- 异步任务失败是否可追踪。
- 错误码是否可读。

### 13.4 数据项

- 迁移是否兼容旧数据。
- 材料格式是否符合脚本要求。
- 题库快照是否保护历史记录。
- 本地生成产物是否不应提交。

## 14. 冲突处理

### 14.1 预防方式

- 小任务及时提交。
- 开始工作前拉取最新主分支。
- 避免多人同时大幅修改同一文档或同一页面。
- 约定接口字段后再并行开发。

### 14.2 解决原则

- 先理解双方改动目的。
- 保留业务有效内容。
- 不直接覆盖他人修改。
- 冲突解决后重新运行相关测试。

## 15. 演示前协作流程

1. 确认主分支代码可启动。
2. 准备演示账号和数据。
3. 检查 provider 配置。
4. 准备 mock 降级方案。
5. 跑一次端到端演示脚本。
6. 截图关键页面。
7. 准备常见问题回答。
8. 确认每位成员能说明自己的模块。

## 16. 贡献记录整理

答辩前可以按以下维度整理贡献：

- 功能模块：负责了哪些页面或接口。
- 技术实现：解决了哪些关键问题。
- 测试质量：补充了哪些测试。
- 文档交付：维护了哪些说明。
- 数据材料：整理了哪些题库和知识库。
- 运维部署：完成了哪些启动和排障工作。

贡献记录应基于真实提交、PR、文档和演示结果，不建议只用口头描述。

## 17. 常见问题

### 17.1 文档提交算不算贡献

算。高质量文档能说明系统边界、降低交接成本、支持验收和答辩。但文档应和项目实际相关，避免无意义堆字数。

### 17.2 是否必须每个功能都有测试

课程项目可以按风险分级。登录、面试、报告、练习、管理员导入等主链路建议覆盖测试或至少有手工验收记录。

### 17.3 多人修改同一文件怎么办

提前分工，尽量按章节认领。修改前同步最新代码，提交前检查 diff，冲突解决后让相关成员复核。

### 17.4 如何避免提交敏感信息

使用 `.env.example` 提供示例，不提交真实 `.env`。提交前检查 `git diff --cached`，确认没有密钥、token、个人隐私数据和本地数据库。

## 18. 最小合并标准

一个变更合并前至少满足：

- 功能或文档目标明确。
- 工作区没有无关文件。
- 相关命令或手工步骤已验证。
- 失败风险已说明。
- 文档和接口契约同步。
- 提交信息可读。
