# 业务流程详解

更新时间：2026-06-28

本文从用户视角和系统视角说明 AI-Interview 的主要业务流程，便于课程答辩、测试验收、后续维护和小组成员交接。

## 1. 文档目标

- 说明普通用户从注册到完成面试报告的完整闭环。
- 说明管理员维护材料、题库和知识库的操作路径。
- 说明关键流程中前端页面、后端接口、数据表和异步任务之间的关系。
- 帮助测试人员根据业务步骤设计用例。
- 帮助答辩人员清楚说明平台价值和技术实现。

## 2. 角色划分

| 角色 | 主要目标 | 常用页面 | 关键数据 |
| --- | --- | --- | --- |
| 游客 | 了解系统、注册或登录 | 欢迎页、登录页、注册页 | 无 |
| 普通用户 | 管理简历、JD、预约、面试、练习、报告 | 首页、简历、岗位、面试、练习、报告 | 用户、简历、JD、面试会话、答题记录 |
| 管理员 | 维护题库、导入材料、检查 provider | 管理页、题库管理、知识库重建 | 材料文件、题库、导入任务、健康检查 |
| 开发者 | 维护接口、数据脚本、测试和部署 | 本地开发环境、OpenAPI、CI | 代码、迁移、测试报告、日志 |

## 3. 总体业务闭环

```text
注册登录
  -> 上传简历
  -> 新增或选择 JD
  -> 创建面试或预约面试
  -> 进行多轮问答
  -> 暂停/恢复或结束面试
  -> 异步生成报告
  -> 查看历史和复盘
  -> 进入专项练习
  -> 根据练习结果补强薄弱知识点
```

平台核心不是单次问答，而是把简历、岗位、知识库、面试过程、报告和练习串成持续改进闭环。

## 4. 注册与登录流程

### 4.1 用户操作

1. 用户进入欢迎页或登录页。
2. 新用户打开注册页面，填写邮箱、密码和显示名称。
3. 已注册用户使用邮箱和密码登录。
4. 登录成功后进入首页，前端保存 access token 和 refresh token。
5. 后续业务请求自动携带 `Authorization: Bearer <access_token>`。

### 4.2 系统处理

1. 后端校验邮箱格式和密码复杂度。
2. 注册成功后创建 `user_accounts` 记录。
3. 登录时校验 PBKDF2 密码哈希。
4. 登录成功后签发 access token 和 refresh token。
5. refresh token 只保存哈希，退出登录时撤销。

### 4.3 相关接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/v1/auth/register` | 注册账号 |
| POST | `/api/v1/auth/login` | 登录 |
| POST | `/api/v1/auth/refresh` | 刷新 token |
| POST | `/api/v1/auth/logout` | 退出登录 |
| GET | `/api/v1/auth/me` | 获取当前用户 |

### 4.4 异常场景

- 邮箱已注册：返回明确错误提示，前端引导登录或找回密码。
- 密码过短或复杂度不足：前端和后端都应给出提示。
- access token 过期：前端自动调用 refresh。
- refresh 失败：前端清理登录状态并跳转登录页。

## 5. 简历管理流程

### 5.1 用户操作

1. 用户进入简历管理页面。
2. 点击上传简历，选择 PDF、DOC 或 DOCX 文件。
3. 上传后查看简历列表。
4. 创建面试时选择某一份简历。
5. 对不再使用的简历执行删除。

### 5.2 系统处理

1. 前端使用 `multipart/form-data` 上传文件。
2. 后端保存文件元数据和解析文本。
3. 简历与当前用户绑定，列表只返回用户自己的数据。
4. 如果某简历已被历史面试引用，删除时需要保护，避免历史报告不可复现。

### 5.3 数据落点

| 数据 | 说明 |
| --- | --- |
| `resumes` | 保存简历文件信息、解析文本和用户归属 |
| 文件存储 | 保存原始上传文件 |
| 面试会话快照 | 面试创建时保存必要上下文，保证复盘稳定 |

### 5.4 验收要点

- 只允许登录用户上传。
- 用户不能看到其他用户简历。
- 非法格式文件应被拒绝。
- 上传成功后可以在面试准备页面选择。
- 删除已绑定历史面试的简历时应有保护或明确提示。

## 6. JD 与公司流程

### 6.1 用户操作

1. 用户进入岗位管理页面。
2. 选择系统预置公司或新增自定义 JD。
3. 填写岗位名称、岗位方向和 JD 正文。
4. 创建面试时选择目标 JD。
5. 面试报告根据 JD 和简历匹配情况生成建议。

### 6.2 系统处理

1. 公司列表通过 `/companies` 返回。
2. 用户 JD 写入 `job_descriptions`。
3. 面试创建时保存 JD 标题和正文快照。
4. 历史面试和报告不依赖后续 JD 编辑结果。

### 6.3 设计意义

- 简历提供候选人背景。
- JD 提供岗位要求。
- 面试问题可以围绕岗位方向和候选人经历生成。
- 报告可以评价简历与岗位的匹配度。

## 7. 面试准备流程

### 7.1 用户输入

创建面试前，用户通常需要选择：

- 简历。
- 岗位方向，例如 Java、Web 或通用方向。
- 难度，例如简单、中等或困难。
- 输入模式，例如文本或音频。
- 输出模式，例如文本或语音。
- 题型组合，例如技术、项目、场景、行为。
- 可选 JD。
- 可选面试官语气。

### 7.2 校验规则

- 简历不能为空。
- 题型组合至少选择一种。
- 音频输入需要浏览器权限和后端 ASR provider 支持。
- 语音输出需要 TTS provider 可用。
- JD 为空时仍可基于岗位方向生成通用问题。

### 7.3 创建结果

创建成功后系统返回：

- 面试会话 ID。
- 会话状态。
- 第一题。
- 当前阶段。
- provider 相关状态。

## 8. 面试问答流程

### 8.1 文本问答

```text
用户输入答案
  -> 前端 POST /interviews/{id}/turns
  -> 后端创建 turn job
  -> worker 处理检索和生成
  -> 前端轮询 job
  -> 展示下一题和反馈
```

### 8.2 音频问答

```text
用户录音
  -> 前端上传音频文件
  -> ASR 转写文本
  -> 进入同样的文本问答链路
  -> 可选 TTS 生成语音回复
```

### 8.3 轮次数据

| 字段类型 | 说明 |
| --- | --- |
| 用户回答 | 文本答案或 ASR 转写结果 |
| 当前阶段 | 自我介绍、项目追问、技术追问、场景题等 |
| 下一题 | LLM、规则模板或题库检索结果 |
| pipeline 元数据 | provider、耗时、降级状态、trace ID |

### 8.4 用户体验要求

- 提交答案后立即显示处理中状态。
- 轮询失败时给出重试入口。
- provider 降级时流程仍应继续。
- 历史问答不应因为刷新页面丢失。

## 9. 暂停、恢复与回放流程

### 9.1 暂停

用户在面试过程中可以暂停，系统保存当前会话状态和已完成轮次。

### 9.2 恢复

用户从暂停列表进入会话，继续下一轮问答。恢复时前端应重新拉取会话状态，避免使用过期本地状态。

### 9.3 回放

面试结束后，用户可以查看历史回放。回放材料通常包含：

- 每轮问题。
- 用户回答。
- 生成的追问。
- 轮次时间。
- provider 与降级信息。

### 9.4 价值

暂停和回放能力让系统更接近真实训练工具，而不是一次性聊天页面。用户可以分阶段完成训练，也可以复盘自己的回答结构。

## 10. 报告生成流程

### 10.1 触发条件

用户点击结束面试后，后端触发报告生成任务。

### 10.2 处理过程

```text
POST /interviews/{id}/finish
  -> 写入面试结束状态
  -> 创建报告任务
  -> ReportWorker 汇总简历、JD、轮次和评分依据
  -> 调用 LLM 生成报告
  -> 失败时回退到规则报告
  -> 保存 interview_reports
```

### 10.3 报告内容

报告通常包括：

- 总体评价。
- 优势分析。
- 风险点和薄弱点。
- 岗位匹配度。
- 回答结构建议。
- 后续练习建议。

### 10.4 状态说明

| 状态 | 说明 | 用户可执行操作 |
| --- | --- | --- |
| `GENERATING` | 正在生成 | 等待或刷新 |
| `READY` | 已生成 | 查看详情 |
| `FAILED` | 生成失败 | 重试 |

## 11. 题库练习流程

### 11.1 用户目标

题库练习用于补齐面试报告中暴露的知识短板。用户可以按岗位方向、题型或难度进行专项训练。

### 11.2 基本流程

1. 用户进入练习准备页。
2. 选择题目范围。
3. 系统创建练习会话。
4. 用户逐题作答。
5. 系统保存答案并计算结果。
6. 用户结束练习，查看记录和正确率。

### 11.3 数据快照

练习开始时应保存题目快照，避免管理员后续修改题库导致历史练习记录发生变化。

### 11.4 验收要点

- 练习会话可以完整开始、答题、结束。
- 刷新页面后答题进度可恢复。
- 历史记录能展示题目、答案和解析。
- 管理员更新题库不影响已完成记录。

## 12. 编程练习流程

### 12.1 用户目标

编程练习用于训练算法、数据结构和工程编码能力，补足问答型面试无法覆盖的代码能力评价。

### 12.2 基本流程

1. 用户进入编程练习列表。
2. 选择题目并创建练习会话。
3. 在编辑器中编写代码。
4. 保存草稿。
5. 运行自测。
6. 正式提交。
7. 查看提交结果和历史记录。

### 12.3 系统处理

- 题目来自内置 JSON 材料。
- 草稿写入 `coding_drafts`。
- 提交写入 `coding_submissions`。
- 本地执行服务负责运行测试用例。

### 12.4 风险提示

本地代码执行不等同于强沙箱。课程演示和本地可信环境可以使用，但面向公网的不可信用户需要引入独立沙箱、资源限制和隔离机制。

## 13. 管理员题库维护流程

### 13.1 操作目标

管理员负责维护 Java、Web 和编程练习材料，保证普通用户练习和面试检索有稳定数据来源。

### 13.2 常见操作

- 查看题库列表。
- 新增题目。
- 编辑题干、类别和解析。
- 上传材料文件。
- 触发材料校验。
- 触发题库重建。
- 查看任务报告。

### 13.3 后台处理

管理员操作通常会影响：

- `backend/assets/material/`
- `backend/assets/data/normalized/`
- `backend/assets/data/reports/`
- SQLite 中的题库表。

### 13.4 幂等与并发

导入类操作耗时较长，前端会携带幂等键，后端需要避免重复启动全量任务。管理员界面也应展示任务执行中状态。

## 14. 知识库重建流程

### 14.1 触发场景

- 新增或修改知识库 Markdown。
- 大批量导入题库材料。
- 调整分块策略。
- 更换 embedding provider。
- 演示前需要保证向量库与材料一致。

### 14.2 执行步骤

```text
材料校验
  -> 材料归一化
  -> 知识分块
  -> embedding 生成
  -> 写入 Chroma 数据目录
  -> 生成构建报告
```

### 14.3 成功标准

- 脚本无异常退出。
- 报告文件生成。
- 向量库目录存在。
- 检索评估返回合理结果。
- 面试问答能引用更新后的知识内容。

## 15. Provider 健康检查流程

### 15.1 检查对象

- LLM provider。
- ASR provider。
- TTS provider。
- Embedding 或检索链路。

### 15.2 用户可见影响

| Provider | 不可用影响 | 降级方式 |
| --- | --- | --- |
| LLM | 问题生成质量下降 | 规则模板或 mock |
| ASR | 音频输入不可用 | 改用文本输入 |
| TTS | 语音回复不可用 | 改用文本输出 |
| Embedding | 检索上下文缺失 | 使用题库或模板 |

### 15.3 展示建议

管理员页面展示 provider 状态，普通用户页面展示简化提示，避免把内部错误直接暴露给最终用户。

## 16. 历史记录与复盘流程

### 16.1 历史面试

历史面试用于查看已完成或曾暂停的面试，包括面试名称、岗位方向、时间、状态和报告入口。

### 16.2 练习记录

练习记录用于查看用户在题库练习和编程练习中的表现，便于判断知识点掌握情况。

### 16.3 复盘建议

用户可以按照以下顺序复盘：

1. 先看报告总体评价。
2. 再看低分项对应的问答轮次。
3. 提炼回答中缺少的关键词、数据和结构。
4. 进入题库练习补足基础知识。
5. 对项目题和场景题重新组织 STAR 或问题-行动-结果结构。

## 17. 端到端验收主线

推荐演示时使用以下主线：

1. 注册新用户。
2. 登录系统。
3. 上传简历。
4. 新增 JD。
5. 创建文本面试。
6. 完成三轮问答。
7. 暂停并恢复面试。
8. 结束面试。
9. 查看报告生成状态。
10. 打开报告详情。
11. 进入题库练习。
12. 完成一次练习并查看记录。
13. 管理员登录。
14. 查看 provider 健康检查。
15. 触发材料校验或题库导入 dry-run。

## 18. 常见业务问题

### 18.1 没有真实模型能否演示

可以。系统支持 mock 和规则回退，重点展示业务闭环、数据流、接口设计和降级策略。

### 18.2 为什么需要 JD

JD 让面试不只是通用问答，而是围绕目标岗位评价候选人与岗位要求的匹配度。

### 18.3 为什么需要报告异步生成

报告可能需要调用 LLM 并处理较长上下文，同步阻塞会影响体验。异步任务可以提供生成中状态、失败重试和降级报告。

### 18.4 为什么需要题库练习

面试报告指出问题，题库练习提供后续训练路径，形成“评估-反馈-练习-再评估”的闭环。

## 19. 交接检查清单

- 是否能说明注册登录和 token 刷新流程。
- 是否能说明简历、JD、面试和报告之间的数据关系。
- 是否能演示一次完整面试。
- 是否能说明 provider 不可用时的降级方式。
- 是否能说明管理员如何维护题库和知识库。
- 是否能找到 OpenAPI、Postman、测试脚本和部署手册。
- 是否能指出本地演示和生产部署之间的差异。
