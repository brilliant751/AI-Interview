# 数据库字段字典

更新时间：2026-06-28

本文补充说明 AI-Interview 的主要数据表、字段含义、业务关系和维护注意事项。字段名称以当前 SQLite 迁移和 repository 实现为依据，实际完整结构以 `backend/migrations/` 和 `backend/app/repositories/interview_repository.py` 为准。

## 1. 使用说明

- 本文用于答辩、测试设计、开发交接和数据排查。
- 表结构可能随功能迭代变化，新增字段后应同步更新本文。
- SQLite 是当前项目的默认存储，生产化部署时可迁移到 PostgreSQL。
- 若本文与迁移文件不一致，以迁移文件和运行时 schema 为准。

## 2. 数据域总览

| 数据域 | 代表表 | 说明 |
| --- | --- | --- |
| 认证 | `user_accounts`、`auth_refresh_tokens`、`auth_password_reset_tokens` | 用户、令牌、密码重置 |
| 简历 | `resumes` | 简历文件、解析文本和用户归属 |
| 岗位 | `job_descriptions`、`companies` | JD、公司和面试目标 |
| 面试 | `interview_sessions`、`interview_turns`、`interview_turn_jobs` | 会话、轮次和异步生成任务 |
| 预约 | `interview_schedules` | 未来面试安排和日历导出 |
| 报告 | `interview_reports` | 面试报告状态和内容 |
| 幂等 | `request_idempotency` | 写请求防重复提交 |
| 题库练习 | `practice_choice_questions`、`practice_sessions`、`practice_session_questions`、`practice_answers` | 选择题练习 |
| 编程练习 | `coding_questions`、`coding_sessions`、`coding_drafts`、`coding_submissions` | 编程题、草稿和提交 |

## 3. 通用字段约定

| 字段 | 说明 |
| --- | --- |
| `id` | 主键或业务 ID，通常由后端生成 |
| `user_id` | 用户归属，用于数据隔离 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `deleted_at` | 软删除时间，存在时表示记录不再作为默认查询结果 |
| `status` | 当前业务状态 |
| `metadata` | 扩展信息，通常为 JSON 字符串 |

时间字段建议统一使用 ISO 8601 格式保存，前端展示时再转换为本地时间。

## 4. `user_accounts`

### 4.1 表用途

保存系统登录账号、角色和密码哈希。普通用户和管理员都属于该表。

### 4.2 关键字段

| 字段 | 说明 | 维护要求 |
| --- | --- | --- |
| `id` | 用户 ID | 创建后不变 |
| `email` | 登录邮箱 | 应唯一 |
| `password_hash` | 密码哈希 | 不返回给前端 |
| `display_name` | 显示名称 | 可用于页面展示 |
| `role` | 角色 | 区分普通用户和管理员 |
| `is_active` | 是否启用 | 禁用后应拒绝登录 |
| `created_at` | 注册时间 | 自动写入 |
| `updated_at` | 更新时间 | 修改资料时更新 |

### 4.3 安全要求

- 不保存明文密码。
- 不在日志中打印密码、token 或完整哈希。
- 管理员权限判断必须基于后端鉴权结果，不能只依赖前端路由。

## 5. `auth_refresh_tokens`

### 5.1 表用途

保存 refresh token 哈希，实现 access token 过期后的续期和退出登录后的撤销。

### 5.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | token 记录 ID |
| `user_id` | 所属用户 |
| `token_hash` | refresh token 哈希 |
| `expires_at` | 过期时间 |
| `revoked_at` | 撤销时间 |
| `created_at` | 创建时间 |

### 5.3 维护规则

- refresh token 轮换后旧 token 应失效。
- 用户退出登录后应撤销当前 token。
- 发现异常 token 使用时可撤销用户全部 refresh token。

## 6. `auth_password_reset_tokens`

### 6.1 表用途

支持找回密码和重置密码流程。

### 6.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 重置 token 记录 ID |
| `user_id` | 所属用户 |
| `token_hash` | 重置 token 哈希 |
| `expires_at` | 过期时间 |
| `used_at` | 使用时间 |
| `created_at` | 创建时间 |

### 6.3 维护规则

- token 应短期有效。
- 使用后立即标记为已使用。
- 不应在响应或日志中暴露完整 token。

## 7. `resumes`

### 7.1 表用途

保存用户上传简历的元数据、解析文本和文件路径。

### 7.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 简历 ID |
| `user_id` | 上传用户 |
| `filename` | 原始文件名 |
| `content_type` | MIME 类型 |
| `file_path` | 存储路径 |
| `parsed_text` | 解析出的文本 |
| `created_at` | 上传时间 |

### 7.3 业务关系

- 面试会话创建时引用简历。
- 报告生成时会读取简历文本。
- 简历删除应检查是否被历史面试引用。

### 7.4 排查建议

- 上传失败时检查文件格式和大小。
- 解析为空时检查解析服务或文件内容。
- 用户看不到简历时检查 `user_id` 过滤条件。

## 8. `companies`

### 8.1 表用途

保存公司信息，用于 JD 归类和面试准备。

### 8.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 公司 ID |
| `name` | 公司名称 |
| `description` | 公司简介 |
| `created_at` | 创建时间 |

### 8.3 维护规则

- 预置公司可用于演示。
- 用户自定义 JD 不一定必须绑定公司。
- 公司删除前应考虑是否存在关联 JD。

## 9. `job_descriptions`

### 9.1 表用途

保存岗位描述，包括系统预置和用户上传内容。

### 9.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | JD ID |
| `user_id` | 创建用户，系统预置可为空或使用系统标识 |
| `company_id` | 公司 ID |
| `job_role` | 岗位方向 |
| `title` | JD 标题 |
| `content_text` | JD 正文 |
| `file_path` | 上传文件路径 |
| `created_at` | 创建时间 |

### 9.3 业务关系

- 面试创建时可引用 JD。
- 会话中应保存 JD 快照，避免后续修改影响历史报告。
- 删除用户 JD 时应避免破坏历史会话复盘。

## 10. `voice_tone_profiles`

### 10.1 表用途

保存面试官语气配置，用于控制问题生成和反馈风格。

### 10.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 语气配置 ID |
| `name` | 配置名称 |
| `description` | 展示说明 |
| `prompt_hint` | 给生成链路使用的提示 |
| `is_active` | 是否启用 |

### 10.3 使用建议

- 语气配置不应改变评分标准。
- 不应产生攻击性或歧视性面试语言。
- 关闭某配置后，历史会话仍应保留当时使用的快照信息。

## 11. `interview_sessions`

### 11.1 表用途

保存一次模拟面试的主记录，是面试域的核心表。

### 11.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 面试会话 ID |
| `user_id` | 所属用户 |
| `resume_id` | 关联简历 |
| `jd_id` | 关联 JD |
| `session_name` | 会话名称 |
| `job_role` | 岗位方向 |
| `difficulty` | 难度 |
| `input_mode` | 输入模式 |
| `output_mode` | 输出模式 |
| `question_types` | 题型组合 |
| `status` | 会话状态 |
| `current_stage` | 当前阶段 |
| `started_at` | 开始时间 |
| `finished_at` | 结束时间 |
| `created_at` | 创建时间 |

### 11.3 状态示例

| 状态 | 说明 |
| --- | --- |
| `CREATED` | 已创建但未开始或刚初始化 |
| `IN_PROGRESS` | 进行中 |
| `PAUSED` | 已暂停 |
| `FINISHED` | 已结束 |
| `FAILED` | 异常失败 |

### 11.4 业务关系

- 一条会话对应多条 `interview_turns`。
- 一条会话最多对应一份当前报告，也可支持重试覆盖。
- 预约开始后会创建或绑定会话。

## 12. `interview_turns`

### 12.1 表用途

保存面试中的单轮问答，包括问题、用户回答、反馈和下一题。

### 12.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 轮次 ID |
| `interview_id` | 所属面试 |
| `stage` | 当前阶段 |
| `question_text` | 本轮问题 |
| `answer_text` | 用户回答 |
| `feedback_text` | 反馈 |
| `next_question_text` | 下一题 |
| `pipeline_meta` | provider、耗时、降级等元数据 |
| `created_at` | 创建时间 |

### 12.3 维护规则

- 问题和回答应形成稳定历史。
- 音频转写文本也应写入回答字段或对应元数据。
- pipeline 元数据用于排查生成质量和 provider 故障。

## 13. `interview_turn_jobs`

### 13.1 表用途

保存轮次异步任务状态。用户提交答案后，任务负责执行 ASR、检索、LLM 生成和 TTS 等耗时步骤。

### 13.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | job ID |
| `interview_id` | 所属面试 |
| `turn_id` | 对应轮次 |
| `status` | 任务状态 |
| `result_json` | 成功结果 |
| `error_message` | 错误信息 |
| `created_at` | 创建时间 |
| `finished_at` | 完成时间 |

### 13.3 状态示例

| 状态 | 说明 |
| --- | --- |
| `PENDING` | 等待处理 |
| `RUNNING` | 执行中 |
| `SUCCEEDED` | 成功 |
| `FAILED` | 失败 |

### 15.1 表用途

### 15.2 关键字段

- 报告内容不应泄露其他用户数据。

## 16. `request_idempotency`

### 16.1 表用途

保存写请求的幂等响应，避免前端重试或用户重复点击导致重复创建。

### 16.2 关键字段

| 字段 | 说明 |
| --- | --- |
| `idempotency_key` | 前端传入的幂等键 |
| `user_id` | 所属用户 |
| `method` | HTTP 方法 |
| `path` | 请求路径 |
| `response_status` | 缓存响应状态码 |
| `response_body` | 缓存响应体 |
| `created_at` | 创建时间 |

### 16.3 维护规则

- 幂等键应和用户、路径、方法一起判断。
- 不同用户不能复用彼此的幂等缓存。
- 过旧幂等记录可定期清理。

## 26. 数据排查路径

### 26.1 用户无法登录

1. 检查 `user_accounts.email` 是否存在。
2. 检查 `is_active` 是否启用。
3. 检查后端日志是否有密码校验失败。
4. 检查 refresh token 是否过期或被撤销。

### 26.2 简历上传后看不到

1. 检查 `resumes.user_id` 是否为当前用户。
2. 检查文件保存路径是否存在。
3. 检查解析文本是否为空。
4. 检查接口是否按用户过滤。

### 26.3 面试轮次卡住

1. 检查 `interview_turn_jobs.status`。
2. 查看 `error_message`。
3. 检查 provider 健康状态。
4. 检查 `pipeline_meta` 中的 trace ID。

### 26.4 报告一直生成中

1. 检查 `interview_reports.status`。
2. 检查报告 worker 是否启动。
3. 查看后端日志中的 LLM 调用错误。
4. 尝试执行报告重试接口。

### 26.5 练习记录不一致

1. 检查 `practice_session_questions` 快照。
2. 检查 `practice_answers` 是否完整。
3. 检查管理员是否修改过原题。
4. 确认前端展示使用快照而不是实时原题。

## 27. 数据维护建议

- 演示前备份 `backend/assets/data/sqlite/interview.db`。
- 导入材料前先运行校验脚本。
- 修改表结构时新增迁移文件，不直接手动改库。
- 删除材料或题目前确认是否影响历史记录。
- 不把真实用户隐私数据提交到仓库。
- `.env` 和本地数据库不应进入 Git。
